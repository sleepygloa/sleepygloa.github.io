# 시스템 기초 지식
시스템 기반? 애플리케이션을 가동시키기 위해 필요한 하드웨어나 OS, 미들웨어 등과 같은 인프라.

## 시스템 기반의 구성요소
- 기능 요구사항 : 시스템이나 소프트웨어에서 무엇을 할 수 있을지 모아놓은 것.
- 비기능 요구사항 : 시스템 성능, 신뢰성, 확장성, 운용성, 보안 등과같은 기능 요구사항 이외의 것.
- 하드웨어 : 시스템 기반을 구성하는 물리적인 요소.
- 네트워크 : 시스템 이용자가 원격지에서 엑세스 할 수 있도록 서버들을 연결하기 위한 요구사항
- OS(운영체제) : 하드웨어나 네트워크 장비를 제어하기 위한 소프트웨어로, 하드웨어의 리소스나 프로세스를 관리.
- 미들웨어 : 서버 OS 상에서 서버가 특정 역할을 다하기 위한 기능을 갖고 있는 소프트웨어.

## 클라우드와 온프레미스
### 정의
- 온프레미스 : 회사에서 데이터센터를 보유하고 시스템 구축부터 운용까지 모두 수행하는 형태.
- 퍼블릭 클라우드 : 인터넷을 경유하여 불특정 다수에게 제공되는 클라우드 서비스. Iaas/Paas/Saas
- 프라이빗 클라우드 : 특정 기업 그룹에게만 제공되는 클라우드 서비스.


### 클라우드와 온프레미스의 채택
#### 클라우드
- 트래픽 변동이 많은 시스템 
  - 직원용 시스템(근태 관리 시스템, 회계 시스템, 인사 시스템)과 고객용 시스템(예약 시스템, 온라인 쇼핑, 온라인 게임의 백엔드, 동영상 배포)
  - 직원용 시스템의 경우 이용자가 한정되어 있어 트래픽이 예상하기 쉬우나, 고객용 시스템은 예상하기 어렵다. 따라서 트래픽을 쉽게 증설할 수 있는 클라우드 시스템이 좋다.
- 재해가 많은 지역
  - 재해 발생으로 인한 시스템의 잦은 중단이 우려되면, 클라우드 시스템을 이용하는게 좋다.
- 서비스를 빨리 제공받고 싶은 시스템
  - 온프레미스 처럼 물리적인 장치를 구매, 발주 부터 구축까지 걸리는 시간을 절약하고자한다면 클라우드 시스템을 이용하는 것이 좋다.

#### 온프레미스
- 높은 가용성이 요구되는 시스템
  - 클라우드시스템은 시스템가용성을 클라우드업체가 보장하고 있다. 네트워크가 끊어진다거나, 보장되지 않은 문제가 발생시 할 수 있는 것이없다.
- 기밀성이 높은 데이터를 다루는 시스템
  - 데이터 저장에 대한 물리적인 저장장소를 명확히 할 필요가 있을때.
- 특수한 요구사항이 있는 시스템

  - 시스템 운용 설계

# 하드웨어와 네트워크 기초 지식
## 서버 장비
- CPU : 프로그램의 설계나 처리 등을 수행하는 전자회로 부품, 작동주파수가 클수록 연산능력이 높아지고, CPU의 연산회로인 '코어' 수가 많아지면 처리할 수 있는 수가 많아진다.
- 메모리 : CPU 가 직접 엑세스 할 수 잇는 기억장치. 데이터 용량이 크거나 데이터 전송 속도가 고속 일수록 고성능이지만, 서버용으로는 전력 소모가 적고, 오류 처리가 탑재 되어있는 것을 사용한다.
- 스토리지 : 데이터베이스에 기록하는 데이터 등과 같은 영구적인 데이터를 저장하는 디바이스. 스토리지의 용량이나 읽기, 쓰기 속도가 시스템 전체의 응답 시간에 영향을 준다.
- UPS(무정전 전원장치) : 전원 트러블에 의한 순간적인 차단을 막기 위한 장치.
- KVM 스위치 : 여러 대의 서버를 관리하기 위한 장치.


## 네트워크 주소
- MAC 주소(물리 주소/이더넷 주소) : 네트워크 인터페이스 카드나 무선 LAN 칩과 같은 네트워크 부품에 물리적으로 할당되는 48비트 주소, OSI 참조 모델 2계층(데이터 링크 계층) 에서 사용
- IP 주소 : 인터넷이나 인트라넷과 같은 네트워크 연결된 컴퓨터나 네트워크 장비에 할당되는 식별번호.

## OSI 참조 모델과 통신 프로토콜
| OSI 참조모델 | 대표 프로토콜 | 대표 통신 기기 |
|---|---|---|
| 7계층(L7) | 응용 계층 | HTTP, DNS, SMTP, SSH |방화벽, 로드밸런서 |
| 6계층(L6) | 표현 계층 | | |
| 5계층(L5) | 세션 계층 | | |
| 4계층(L4) | 전송 계층 | TCP, UDP | |
| 3계층(L3) | 네트워크 계층 | IP, ICMP | 라우터, L3 스위치 |
| 2계층(L2) | 데이터 링크 계층 | Ethernet | L2 스위치, 브리지 |
| 1계층(L1) | 물리 계층 | | 리피터 허브 |

- 응용 계층 : 웹의 HTTp나 메일 전송을 하는 SMTP 등과 같은 애플리케이션에 특화된 프로토콜
- 표현 계층 : 데이터의 저장 형식이나 압축, 문자 인코딩과 같은 데이터의 표현 형식
- 세션 계층 : 커넥션 확립 타이밍이나 데이터 전송 타이밍을 규정
- 전송 계층 : 데이터 전송을 제어
- 네트워크 계층 : 서로 다른 네트워크 간에 통신을 하기 위한 규정
- 데이터 링크 : 동일한 네트워크 안에 있는 노드 간의 통신을 규정
- 물리 계층 : 통신 장비의 물리적 및 전기적 특성을 규정

## 방화벽
- 패킷 필터형 : 통과하는 패킷을 포트 번호나 IP 주소를 바탕으로 필터링 하는 방법, 예를 들어 80(http)와 443(https) 만 통과해도 좋다 이런식으로 정의 
- 애플리케이션 게이트웨이형 : 패킷이 아니라 애플리케이션 프로토콜 레벨에서 외부와의 통신을 대체 제어 하는 것. 프록시 서버.

## 라우터/레이어 3 스위치
- 라우터 : 2개 이상의 서로 다른 네트워크 간 중계를 하기 위한 통신 장비, 3계층에서 작동
- 레이어 3 스위치 : 라우터 와 비슷하며, 라우팅을 하드웨어로 처리하여 고속처리된다는 것, 연결하는 이더넷 포트수가 많다는 장점이 있다.

# OS
## Linux
Unix 호환 서버 OS. 보안에 뛰어나며 안정적으로 작동된다는 특징을 가지고 있고, 오픈소스이므로 누구나 자유롭게 수정 및 재배포를 할 수 있습니다.
- Linux 커널
  OS의 코어가 되는 부분을 말함.
  메모리관리, 파일시스템, 프로세스관리, 디바이스 제어 등 OS로서 하드웨어나 애플리케이션 소프트웨어를 제어하기 위한 기본적인 기능을 갖고 있는 소프트웨어.
  특히 Android는 Linux 커널 상에 구축되어 있다.
- Linux 배포판
  Linux 커널과 함께 각종 커맨드, 라이브러리, 애플리케이션이 포함되어있다.
  커널 이외의 부분인 사용자공간(userland) 에 커널을 통해 접근해야한다.
  배포판은? Debian, RedHat, Slackware, 등등.. 자주 들어봤을 Linux 버전들을 배포판이라한다.

### Linux 커널
- 디바이스관리
  하드웨어를 디바이스 드라이버 라는 소프트웨어를 이용해 제어함.
- 프로세스관리
  프로세스란 ? 명령을 실행 할 때, 해당 프로그램 파일에 쓰여있는 내용을 읽어들어 메모리상에 전개한후 메모리상의 프로그램을 실행한다.
  실행한 프로세스에는 PID(프로세스ID) 가 붙으며, 이것으로 관리가 된다.
- 메모리관리
  커널은 프로그램/데이터를 물리 메모리에 효율적으로 할당하거나, 사용하던 메모리 영역을 해제하는 역할을 한다.
  메모리의 물리적인 용량을 초과하는 프로그램을 전개할 때는, 보조기억장치에 가상 메모리 영역을 만들어 전개한다. 이런 영역을 스왑(swap) 이라한다. 
  커널은 사용빈도가 높은 프로그램을 메모리로(스왑인) 보내고, 빈도가 적은 프로그램을 보조기억장치로(스왑아웃) 보낸다.
- 쉘 스크립트
  커널을 조작하기 위해 쉘(Shell)을 사용한다.
  - 애플리케이션 실행/정지/재실행
  - 환경변수 관리
  - 명령 이력 관리
  - 명령 실행 결과 표시 및 파일 출력

### Linux 파일 시스템
  - VFS(Virtual File System : 가상파일시스템) 을 이용하여 데이터에 대한 투과 액세스 가능. 어떤 디스크에 어떤 데이터가 있든지, 어디 네트워크에 있는 데이터인지 관계없이 투과? 하여 접근가능하게 해준다. 또, 접근한 각 디바이스를 파일로 취급한다.

### Linux 디렉토리 구성
사실 처음 서버를 구성한다던가, 리눅스를 배우면, 사용하는 폴더는 몇 없지만 (/bin, /etc, /lib, /usr ...), 적어도 어떤식으로 구성되어 관리되는지 알고 있으면 좋다.
  - ```/bin``` : 기본 커맨드
  - ```/boot``` : OS 시작에 필요한 파일
  - ```/dev``` : 디바이스 파일
  - ```/etc``` :  설정 파일
  - ```/home``` : 사용자 홈 디렉토리
  - ```/lib``` : 공유 라이브러리
  - ```/mnt``` : 파일 시스템의 마운트 포인트용 디렉토리
  - ```/media``` : CD/DVD-Rom의 마운트 포인트
  - ```/opt``` : 애플리케이션 소프트웨어 패키지
  - ```/proc``` : 커널이나 프로세스에 관한 정보
  - ```/root``` : 특권 사용자용 홈 디렉토리 
  - ```/sbin``` : 시스템 관리용 마운트
  - ```/srv``` : 시스템 고유 데이터
  - ```/tmp``` : 임시 디렉토리
  - ```/usr``` : 각종 프로그램이나 커널 소스를 놓아두는 디렉토리
  - ```/var``` : 로그나 메일 등 가변적인 파일을 놓아두는 디렉토리

### Linux 보안 기능
 - 계정에 대한 권한 
  - root, 일반사용자
  - root 계정 : 시스템 종료, 파일시스템 마운트/언마운트, 애플리케이션 설치 등의 실행권한
  - 계정그룹 설정과 퍼미션 설정으로 관리 가능
- 네트워크 필터링을 사용한 보안
  - 패킷 필터링 : 패킷이 헤더 부분(송신IP, 수신IP, 포트번호 등)의 조건이 일치하면, 설정한 액션(전송, 파기, 주소수정)을 수행한다.
- SELinux(Security-Enhanced Linux) 
  - 보안 대상에 따라 HTTP/FTP 와 같은 프로세스마다 액세스 제한을 거는 Type Enforcement(TE)와 root도 포함한 모든 사용자에 관해 제어를 거는 롤베이스 액세스 제어(RBAC)로 제어.

# 웹 서버/웹 애플리케이션 서버
웹 서버 ? 클라이언트의 브라우저가 보내온 HTTP 요청을 받아, 웹 콘텐츠를 응답으로 반환하거나, 다른 서버 사이드 프로그램을 호출하는 기능을 갖는 서버.
  - Apache HTTP Server : 오픈소스 웹 서버. 소규모 웹 사이트부터 대규모 업무 시스템 까지 많이 사용
  - Internet Information Services : MS 가 제공하는 웹 서버
  - Ngnix(엔진엑스) : 오픈 소스 웹 서버. 소비 메모리가 적으며 리버스 프록시 기능과 로드밸런서 기능도 갖고 있음.

# 데이터 베이스 서버
시스템이 생성하는 다양한 데이터를 관리하기 위한 미들웨어.
- RDBMS : 데이터 CRUD, 트랜젝션 처리 등을 처리한다. 
- NoSQL : 병렬분산처리나 유연한 스키마 설정. KVS(Key-Value 스토어)나 도큐먼트 지향데이터베이스, XML 데이터베이스 등. 대량의 데이터 축적이나 병렬처리에 뛰어나기 때문에 다수의 사용자 액세스를 처리할 필요가 있는 온라인 시스템에서 널리사용
  - Redis : 메모리상에 KVS를 구축 할 수 있는 OSS 의 NoSql. 정기적으로 백업이 되지만, 갱신할 때마다 기록되는 것이 아니기 때문에, 예상치 못한 전원 차단이 있으면 데이터가 손실되는 경우가 있다.
  - MongoDB : 도큐먼트라는 구조적 데이터를 콜렉션으로 관리하는 OSS. 복잡한 계층 구조를 가지게 할 수 있어서 그러한 구조에 포함되는 필드를 지정한 쿼리나 인덱스의 생성이 가능. 온라인 게임이나 로그 분석 등에 이용
  - Apche Cassandra : 하나의 Key 에 대해 여러 개의 컬럼을 가지고 관계형 테이블에 가까운 데이터 구조를 갖고 있는 OSS. Facebook이 대규모 데이터를 저장하기 위해 개발.

# Docker ?
컨테이너 기술을 사용하여 애플리케이션의 실행 환경을 구축 및 운용하기 위한 플랫폼.

# 컨테이너 기술 개요
## 컨테이너
- 컨테이너란? 호스트 OS상에 논리적인 구획(컨테이너)를 만들고, 애플리케이션을 작동시키기 위해 필요한 라이브러리나 애플리케이션을 하나로 모아, 마치 별도의 서버 인 것처럼 사용 할 수 있게 만드는 것.
- 호스트 OS 리소스를 논리적으로 분리하고 여러 컨테이너가 공유하여 사용한다. 오버헤드가 적기 때문에 속도가 빠르다.
- 물리서버 하나에 여러 애플리케이션이 돌아가는 구조는 디렉토리구조를 공유하고, 서버의 IP 주소를 같이 사용하기 때문에 서로 영향을 미치지않게 조심해야하지만, 컨테이너 기술은 동일한 자원을 사용함에도 각 애플리케이션들이 점유하는 것처럼 보이는 장점이있다.?
- 호스트형 서버 가상화 : Virtural Box, VMWare Workstation Player 같은 os 부터 설치하는 가상화 방법. 접근하기 위해서 Guest 등의 유저를 만들고 가상화OS 로 이뤄진 컨테이너를 사용하기 때문에 무겁다.
- 하이퍼바이저형 서버 가상화 : 가상화전문 S/W '하이퍼바이저'를 배치. Hyper-V, XenServer, 호스트 OS 없이 하드웨어를 직접 제어하기 때문에 자원을 효율적으로 사용.


# Docker 개요
애플리케이션의 실행에 필요한 환경을 하나의 이미지로 모아두고, 그 이미지를 사용하여 다양한 환경에서 애플리케이션 실행 환경을 구축 및 운용하기 위한 오픈소스 플랫폼.

- 애플리케이션의 실행 모듈
- 미들웨어나 라이브러리군
- OS/네트워크 등과 같은 인프라 환경 설정

- 개발 환경 배포 비교
  - 폭포형 애플리케이션 개발을 할 때는, 개발환경 구성 -> 테스트 환경 구성 -> 스테이징 환경 구성 -> 제품 환경 구성. 으로 개발환경을 구성하게 되고, 변경 사항이있을때는, 개발환경부터 차례대로 재구성해야한다.
  - 도커를 사용한다면? 애플리케이션 실행에 필요한 모든 환경을 도커 이미지로 작성. 도커만 깔려있다면 모든 상황에서 동일한 개발환경을 제공하게 된다.

## Docker 이미지를 공유하는 기능(Ship)

Github, BItbucket과 연계. Github 에 Dockerfile 을 관리하고, 이미지를 자동생성하여 자동배포도 가능하다
## Docker 컨테이너를 작동시키는 기능(Run)

# Docker 작동구조
## 컨테이너 구획화(namespace)
- PID namespace : 프로세스에 할당된 고유 ID. PID와 프로세스를 격리하여, 다른 프로세스 끼리는 접근할 수 없다.
- Network namespace : 네트워크 디바이스, IP 주소, 포트 번호, 라우팅 테이블, 필터링 테이블 등 네트워크 리소스로 격리. 호스트 OS 에서 사용중인 포트 번호도, 컨테이너안에서 사용가능.
- UID namespace : UID(사용자ID), GID(그룹ID) 별로 격리. 이 격리된 컨테이너안에서 호스트OS 와는 별개의 root, 일반 사용자를 사용할수 있다.
- MOUNT namespace : 파일 시스템을 사용하기 위한 마운트를 격리. 다른 namespace, 호스트 OS 에서 접근 할 수 없다.
- UIS namespace : 호스트명이나, 도메인 명을 독자적으로 가질 수 있다.
- IPC namespace : 프로세스간의 통신(IPC) 오브젝트의 격리. 

## 릴리스 관리 장치(cgroups)
- 프로세스와 스레드를 그룹화 하여, 그 그룹안에 존재하는 프로세스와 스레드에 대한 관리를 수행하기 위한 기능
- 컨테이너 안의 프로세스에 대해 제한을 함으로 어떤 컨테이너가 호스트OS 자원을 모두 사용하는 것을 막음.
  - cpu : CPU 사용량을 제한
  - cpuacct : CPU 사용량 통계 정보를 제공
  - cpuset : CPU나 메모리 배치를 제어
  - memory : 메모리나 스왑 사용량을 제한
  - devices : 디바이스에 대한 액세스 허가/거부
  - freezer : 그룹에 속한 프로세스 정지/재개
  - net_cls : 네트워크 제어 태그를 부가
  - bikio : 블록 디바이스 입출력량 제어

  ## 네트워크 구성(가상 브리지/가상 NIC)
  - 도커 실행 시 서버 물리NIC가 docker0(가상 브리지)로 생성, 프라이빗IP(172.17.0.0/16, 서브넷 마스크)가 eth0으로 생성.
  - 가상 NIC 는 OSI 참조모델 레이어 2 로 가상 NIC 간 터널링 통신.
  - 물리NIC 와 가상NIC 간 연결하기 위해 NAPT 사용

  ### NAPT(Network Address Port Translation)
  프라이빗 IP 주소와 함께 포트번호와 같이 변환 하는 기술.
  프라이빗 IP 주소를 글로벌 IP 주소로 변환 할 때 프라이빗 IP 주소별로 서로 다른 포트 번호로 변환.

  ## Docker 이미지의 데이터 관리 장치
  - AUFS : 다른 파일 시스템의 파일이나 디렉토리를 투과적으로 겹쳐서 하나의 파일 트리를 구성할 수 있는 파일시스템.
  - Btrfs : Copy on Write. 롤백과 스냅샷 기능
  - Device Mapper : 파일 시스템의 블록 I/O와 디바이스의 매핑 관계를 관리
  - OverlayFS : 파일 시스템에 다른 파일 시스템을 투과적으로 merging하는 장치.
  - ZFS :  볼륨 관리, 스냅샷, 체크섬 관리, 리플리케이션 등 지원.
